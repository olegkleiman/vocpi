## What This Is

VOCPI is a FastAPI-based eMSP (e-Mobility Service Provider) server implementing OCPI 2.2.1 (Open Charge Point Interface) for EV charging in Israel. It acts as the intermediary between a mobile app/frontend and CPO (Charge Point Operator) systems.

## Commands

### Run the server
```bash
uvicorn src.main:app --host 0.0.0.0 --port 8080 --reload
```

### Run tests
```bash
pytest -v tests/
# Single test:
pytest -v tests/test_foo.py::test_bar
```

### Install dependencies
```bash
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

## Environment Variables

Copy `.env.example` to `.env`. Required vars:

| Variable | Description |
|---|---|
| `PG_HOST`, `PG_PORT`, `PG_DB`, `PG_USERNAME`, `PG_PASSWORD` | PostgreSQL connection |
| `PG_USE_SSL` | SSL mode (`disable`, `require`, etc.) |
| `RFID_FAKE_TOKEN_FILE_PATH` | Path to JSON file with RFID token for START_SESSION commands |
| `CALLBACK_BASE_URL` | Base URL this server is reachable at (for CPO callbacks) |
| `LOG_LEVEL` | Logging level (default: `INFO`) |
| `LOCATION_REFRESH_TIMEOUT_SEC` | SSE polling timeout for location updates (default: `30.0`) |

## Architecture

### URL Structure

- **OCPI endpoints**: `/ocpi/2.2.1/...` ‚Äî standard OCPI protocol routes consumed by CPO partners
- **Custom API**: `/api/...` ‚Äî internal routes consumed by the mobile app / frontend

### Request Flow

**Session lifecycle (typical)**:
1. Frontend calls `POST /api/begin_session` ‚Üí creates a `session_request_id`, sends `START_SESSION` to CPO via OCPI
2. CPO calls back `POST /ocpi/2.2.1/sessions/` ‚Üí eMSP saves the session, publishes to in-memory pub/sub
3. Frontend subscribes `GET /ocpi/2.2.1/sessions/updates/{session_request_id}` (SSE) for real-time updates
4. Frontend calls `POST /api/end_session` ‚Üí sends `STOP_SESSION` to CPO
5. CPO calls back `POST /ocpi/2.2.1/cdrs` ‚Üí CDR saved; long-polling waiter in `GET /api/cdrs/updates/{session_id}` is released

### Key Design Patterns

**Dual-router pattern** (`src/router.py`): Each command module exports two routers ‚Äî `router` (OCPI standard routes) and `api_router` (custom frontend routes). Both are registered in `src/main.py`.

**`sessions_requests` table as correlation table**: The `request_id` (generated by the frontend) is the stable identifier throughout a session lifecycle. The CPO's `session_id` is associated with it after the CPO calls back. `SessionService` methods map between these two IDs.

**In-process pub/sub** (`src/pubsub.py`): `OCPIPubSub` is a singleton (created in `src/dependencies.py`) that holds `asyncio.Queue` per topic. Topics are:
- `{session_request_id}` ‚Äî session update events
- `{location_id}:{evse_id}` ‚Äî location/EVSE status events

**SSE vs long-polling**: Session and location updates use SSE (`sse-starlette`). CDR receipt uses HTTP long-polling via `asyncio.Event` in `cdr_waiters` dict.

**Partner lookup with caching**: `get_partner()` in `database.py` and `SessionService.get_partner()` are decorated with `@cached(ttl=600)` from `aiocache`. Partner data (base_url, token, version) is resolved from `ocpi_locations` ‚Üí `ocpi_evses` ‚Üí `ocpi_partners`.

### Models

Three model layers exist:
- `src/models/ocpi/models_ocpi.py` ‚Äî OCPI 2.2.1 protocol models (wire format from CPO)
- `src/models/sqlalchemy/models.py` ‚Äî DB ORM models
- `src/models/pydantic/models.py` ‚Äî Internal/API models (frontend wire format)

### Database

PostgreSQL with async SQLAlchemy (psycopg3). Connection pool: 5 permanent + 10 overflow, recycled every 5 minutes. Migrations are plain SQL files in `src/sql/migrations/` and queries in `src/sql/queries/`.

### Services (`src/app_services/`)

- `SessionService` ‚Äî CRUD for sessions, session requests, and the request_id ‚Üî session_id mapping
- `LocationService` ‚Äî fetches location details from CPO via OCPI HTTP and converts to frontend model
- `TariffService` ‚Äî tariff lookup
- `CDRService` ‚Äî saves and retrieves Charge Detail Records

### Deployment

CI/CD via Bitbucket Pipelines ‚Üí Docker ‚Üí AWS ECR ‚Üí ECS. The `main` branch deploys to production; `develop` deploys to development. Tags (`v*`) trigger ECR pushes with version tags.
